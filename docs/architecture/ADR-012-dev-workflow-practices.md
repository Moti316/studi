# ADR-012: קודיפיקציה של נוהלי-פיתוח — commit-בסיום-משימה · doc-discipline · single-branch · שער-בדיקות

> **Status**: Proposed

> **Authors**: studio-lead · motilev8
> **Phase**: 0

>
> מטרה: לעגן בכתב את נוהלי-העבודה היומיומיים שכבר מבוצעים בפועל, כדי שכל סוכן/בני-אדם
> יעבוד אותו-דבר. מקור: החלטות-מוטי החוזרות (commit+push לכל משימה-ירוקה · single-branch

> **מעודכן: 2026-06-01**

## Context (למה צריך החלטה)

הפרויקט מתנהל ע"י יוצר-יחיד (מוטי) בעזרת צוות סוכני-AI. בלי נוהלי-עבודה כתובים, כל סוכן עלול
לבחור דפוס שונה: מתי לעשות commit, אם לפתוח ענף, אילו מסמכים לעדכן כשנוסף .md חדש, ואיזה
שער-איכות לעבור לפני commit. התוצאה היא drift — היסטוריית-git לא-עקבית, מסמכים שנכתבים אך לא
נרשמים באינדקסים, ודחיפות שנכשלות בגלל הגדרת-git שגויה.

האילוצים בפועל כבר התגבשו והם חוזרים בכל סשן:

- מוטי דורש **commit + push מיידי בסיום כל משימה-ירוקה** — לא לצבור עבודה לא-מגובה.
- הריפו עובד **single-branch על `main`** (push ישיר), באישור-מוטי כפול — אין ערך ב-PR-flow
  לעבודת-יוצר-יחיד.
- **משמעת-תיעוד**: כל קובץ `.md` חדש חייב להירשם בזיכרון + `CLAUDE.md` + `PROJECT-MAP.md`,
  אחרת מסמכים "נעלמים".
- דחיפות נכשלו בעבר עם `GH007` בגלל אימייל-commit שגוי → חייב אימייל-`noreply`.

ההחלטה הזו רק **מקודדת מצב-קיים** — היא לא מציגה תהליך חדש ולא דורשת התקנת כלים.

## Decision (מה הוחלט)

לעגן חמישה נוהלי-עבודה כחוזה-מחייב לכל סוכן ולכל בני-אדם בפרויקט. כולם כבר מבוצעים בפועל;
ה-ADR נותן להם מעמד-רשמי וניתן-לציטוט.

**Pattern שנאמץ**: "סיום-משימה = commit ירוק + push ל-`main` + עדכון-אינדקסים".
**מה זה מחליף**: נהלים בעל-פה / מובלעים שהתפזרו בין הזיכרון ל-`CLAUDE.md`.

### 1. Commit + push בסיום כל משימה-ירוקה

בסיום כל יחידת-עבודה שעוברת את שער-הבדיקות (להלן) — `git add` ממוקד, commit עם הודעה-תיאורית,
ו-`push` מיידי. אין לצבור עבודה לא-דחופה בין משימות.

### 2. משמעת-תיעוד (doc-discipline)

כל קובץ `.md` חדש נרשם **באותו commit** בשלושה מקומות:

| יעד-רישום                     | מה נרשם                               |
| ----------------------------- | ------------------------------------- |
| זיכרון (`MEMORY.md` + פתק)    | רשומה קצרה + לינק לפתק-ייעודי אם נדרש |
| `CLAUDE.md`                   | שורת-הפניה למסמך החדש                 |
| `docs/context/PROJECT-MAP.md` | מיקום המסמך במפת-הפרויקט              |

ADRs מקבלים בנוסף שורה ב-`docs/architecture/README.md`. כל קובץ `.md` נשמר prettier-formatted
(printWidth 100).

### 3. Single-branch `main` (push ישיר, ללא ענפים)

עבודה ישירה על `main` ו-push ישיר אליו. **אין פתיחת ענפים** ואין PR-flow. אושר ע"י מוטי
פעמיים. מתאים למודל יוצר-יחיד שבו אין מיזוג-מקביל ואין reviewer חיצוני.

### 4. שער typecheck + test לפני commit

לפני כל commit שנוגע בקוד או במסמכי-תוכן יש להריץ ולוודא-ירוק:

- `prettier --write` על הקבצים שנגעו בהם (או `pnpm lint:fix`).
- `pnpm typecheck` (`tsc --noEmit`) — חייב לעבור.
- `pnpm test` (`vitest run`) — חייב להישאר ירוק; במיוחד אינוריאנט ה-57-scope.

commit אדום לא נדחף.

### 5. אימייל-git = `noreply`

כל commit חתום באימייל-`noreply` של GitHub
(`223746285+Moti316@users.noreply.github.com`). אימייל אחר גורם לכשל-דחיפה `GH007`.

**מקרא**: ✅ נוהל-מבוצע-וקודד · 🟠 קיים-חלקית-ומעוגן-עכשיו · 🔴 לא-רלוונטי/נדחה.

| נוהל                    | סטטוס | מקור-ההחלטה      |
| ----------------------- | ----- | ---------------- |
| commit+push בסיום-משימה | ✅    | החלטת-מוטי חוזרת |
| משמעת-תיעוד             | ✅    | כלל-זיכרון קיים  |
| single-branch `main`    | ✅    | אישור-מוטי ×2    |
| שער typecheck+test      | ✅    | נוהל-בדיקה קיים  |
| אימייל `noreply`        | ✅    | מניעת `GH007`    |

## Alternatives Considered (לא בחרנו)

### Option A: Feature-branch + Pull-Request flow

- ✅ מקובל בצוותים רב-משתתפים, מאפשר code-review רשמי
- ✅ היסטוריה נקייה עם merge-commits ממוקדים
- ❌ overhead מיותר ליוצר-יחיד — אין reviewer חיצוני, אין מיזוג-מקביל
- ❌ מוטי אישר במפורש single-branch ×2; PR-flow מנוגד להחלטה

### Option B: commit מקובץ בסוף-סשן (batch)

- ✅ פחות commits, היסטוריה "תמציתית"
- ❌ עבודה לא-מגובה נצברת — סיכון-אובדן אם הסשן נקטע
- ❌ commit ענק מקשה על bisect ועל הבנת-שינוי נקודתי

### Option C: gstack (git-stacked-branches)

- ✅ מאפשר ערימות-ענפים תלויות לפיתוח-מקבילי
- ❌ **נדחה: רעיון-בלבד, לא מתקינים.** הכלי לא מותקן ולא ייצור צעד-התקנה בשום מקום.
  מנוגד ל-single-branch (החלטת-מוטי) ומיותר ליוצר-יחיד.

**למה לא A/B/C**: A ו-C מנוגדים להחלטת single-branch של מוטי; B מסכן עבודה לא-מגובה. נוהל
ה-commit-המיידי על `main` הוא הפשוט והבטוח ביותר למודל יוצר-יחיד עם סוכני-AI.

## Consequences

### Positive

- כל סוכן עובד אותו-דבר — אין drift בהיסטוריית-git ובאינדקסי-המסמכים
- עבודה מגובה תמיד (push מיידי) — אפס-סיכון-אובדן בין משימות
- מסמכים לא "נעלמים" — doc-discipline מבטיח שכל .md חדש נמצא בכל האינדקסים
- שער-הבדיקות שומר על `main` ירוק-תמיד

### Negative / Trade-offs

- היסטוריית-git עם commits רבים-וקטנים (מקובל ליוצר-יחיד; bisect דווקא נוח יותר)
- אין code-review רשמי — מתקזז ע"י שער typecheck+test ובדיקת-עצמי
- single-branch לא יתאים אם יצטרף מפתח שני; אז יידרש ADR-משלים (migration-path: לעבור
  ל-PR-flow ולסמן ADR זה כ-Superseded)

### Neutral

- אין שינוי-תהליך מעשי — כל הנהלים כבר מבוצעים. ה-ADR רק מתעד ומאפשר ציטוט.
- אין כלי חדש להתקין.

## Implementation Plan

הנהלים כבר מבוצעים. הצעדים הבאים הם **רישום וקיבוע**, לא התקנה חדשה:

1. לעגן את חמשת הנהלים כסעיף ב-`CLAUDE.md` (קישור ל-ADR זה).
2. לרשום ADR זה ב-`docs/architecture/README.md` ובמפת-הפרויקט (`PROJECT-MAP.md`).
3. להמשיך להריץ את שער typecheck+test לפני כל commit (כפי שכבר נעשה).
4. לוודא שהגדרת `git config user.email` = אימייל-`noreply` (כפי שכבר מוגדר).
5. לשמור על push ישיר ל-`main` ללא פתיחת ענפים (כפי שכבר נעשה).

**שעות עבודה מוערכות**: 0 (רישום בלבד — אין כלי חדש; אין צעד-התקנה).
**Phase שבו זה ייעשה**: 0 (מיידי, חופף לעבודה הקיימת).
