## עיקרון-על

איכות = **שער חוסם (blocking gate)**, לא המלצה. תוצר חוצה משער רק כשהוא ירוק.
חמשת השערים פועלים בשתי רמות:

- **שערים 1-3 (אוטומטיים, מכניים):** רצים ב-git-hook / CI על כל commit. מהירים, דטרמיניסטיים.
- **שערים 4-5 (ממשליים, סוכניים):** drift + oversight — שומרים שהסוכנים עצמם נשארים בתוך תפקידם.

עיקרון `{{APPROVAL_CONDITION}}`: דחיפה ל-`{{DEFAULT_BRANCH}}` מותרת רק כששער 1+2+3 ירוקים
ואין דגל-סחף פתוח (שער 4) או צו-עצירה פעיל (שער 5).

## שער 1 — Lint / Format (pre-commit)

**מתי:** על כל commit (אוטומטי), ובמלואו ב-CI.
**מטרה:** סגנון-קוד אחיד, ללא שגיאות-lint, פורמט עקבי.

**מימוש מומלץ (אופציה, לא תלות-קשיחה):** hook קדם-קומיט שמריץ lint רק על קבצים-מבוימים
(staged) כדי לשמור מהירות. ניתן לממש עם כל מנגנון-hook (למשל husky, lefthook, או
`.git/hooks` ידני) — husky היא **אחת מהאופציות**, לא דרישה.

דוגמת hook קדם-קומיט גנרית:

```sh
# {{PKG_MANAGER}} — הרצת lint על קבצים-מבוימים בלבד
{{PKG_MANAGER}} exec lint-staged

# (אופציונלי) רענון-אוטומטי של נכס-נגזר ו-restage אם השתנה. non-fatal.
node <generated-asset-script> && git add <generated-asset> || true
```

תצורת `lint-staged` לדוגמה (פר-סוג-קובץ):

```json
{
  "lint-staged": {
    "*.{<code-ext>}": ["<linter> --fix", "<formatter> --write"],
    "*.{md,json,yaml}": ["<formatter> --write"]
  }
}
```

**שער מלא ב-CI (לא רק staged):**

```bash
{{LINT_CMD}}
```

ברירת-מחדל ל-`{{LINT_CMD}}` אם לא הוגדר: `<linter> . && <formatter> --check "**/*.{<code-ext>,md,json}"`.

**קריטריון-מעבר:** 0 שגיאות-lint, 0 חריגות-פורמט. אזהרות מתועדות אך אינן חוסמות
(אלא אם הפרויקט הגדיר אחרת).

**מתי:** על כל commit (lite), ובמלואו ב-CI לפני merge ל-`{{DEFAULT_BRANCH}}`.
**מטרה:** כל הבדיקות עוברות; TDD-first (בדיקה-כושלת לפני הקוד).

## שער 4 — Drift (בקרת-סחף סוכנית)

> סנתז מ-§בקרת-סחף ב-`{{TEAMS_DIR}}/ORG.md`. שער זה אינו מכני — הוא שומר שהסוכנים
> נשארים בתוך ה-`identity.md` שלהם ולא חורגים מ-Scope/Red-Lines.

**מהו סחף (drift):** סוכן שמתחיל לפעול מחוץ ל-`identity.md` שלו — חורג מ-Scope,
מתעלם מ-Red-Line, ממציא תוכן (קביעות נגד {{PRIMARY_SOURCE_OF_TRUTH}}), או חוזר על אותה שגיאה.

### 4א. Self-check (חיישן-עצמי, שדה-חובה)

בסיום כל משימה, הסוכן ממלא ב-`activity-log.md` שלו את שדה ה-**Self-check**:

> "האם נשארתי בתוך Scope+Red-Lines? כן/לא + נימוק."

`Self-check = לא` → דגל-סחף אוטומטי.

### 4ב. שלושת החיישנים לאיתור-סחף

1. **Self-check נכשל** — שדה ה-Self-check ב-`activity-log.md` מסומן "לא".
2. **שער-צוות** — `<lead>` (ראש-הצוות) מזהה אי-התאמה בין הדיווח ל-`identity.md`.
3. **loop-breaker** — אותה error-hash {{DEBATE_MODELS_N}}-פעמים-רצופות (ברירת-מחדל: 3)
   → `CRITICAL STOP`. (מתואם עם guardrails בקובץ-הסוכנים הקנוני.)

> הערה: סף ה-loop-breaker פרמטרי. אם הפרויקט לא הגדיר ערך ייעודי, השתמש ב-**3**.

### 4ג. סולם-טיפול (Escalation)

1. `<lead>` מסמן **דגל** ב-`control-report.md` (slug + סוג-הסטייה + ראיה).
2. **the orchestrator** מכריע: **שיקום** (הסוכן מתוקן, ה-`identity.md` מובהר, ממשיך)
   מול **החלפה** (הסוכן מארכב ל-`_archive/` → סוכן-חדש עם `memory.md` נקי).
3. הפרת-Red-Line קריטית (secret שדלף, תוכן מומצא שפורסם) → **the orchestrator** עוצר
   מיידית את הסוכן ומעלה ל-Council (`{{COUNCIL_MEMBER}}`).

**קריטריון-מעבר:** אין דגל-סחף פתוח שלא הוכרע. דגל-Red-Line קריטי פתוח = חוסם-מוחלט.

## שער 5 — Oversight (צו-עצירה קוורום)

> סנתז מ-`{{TEAMS_DIR}}/oversight/_oversight-protocol.md`. ענף-בקרה עצמאי שמדווח
> **ישירות ל-Council** (`oversight → council`), עוקף את the orchestrator.
> מופעל רק אם `{{ENABLE_OVERSIGHT}} = true`.

**מהות:** ביקורת בלתי-תלויה (read-only על תוצרי-הביצוע). אינה בונה — רק מבקרת,
ומחזיקה סמכות **צו-עצירה (stop-order)**.

### 5א. תנאי-הפעלה

ראש-ענף-הבקרה (`<oversight-lead>`) או אחד ממבקרי-הענף מזהה הפרה מהותית
(סטייה-מתוכנית, כיסוי-חסר, תוכן מחוץ-ל-scope, סיכון-אבטחה/פרטיות) → יוזם צו-עצירה.

### 5ב. קוורום

צו-עצירה תקף רק בחתימת-קוורום **{{OVERSIGHT_QUORUM}}** (ברירת-מחדל: 2/3 ממבקרי-הענף)
**+ ראיה ב-ledger** (`stop-orders-ledger.md`). ללא קוורום + ראיה — אין צו.

### 5ג. תוצאה

צו-עצירה פעיל **חוסם** את ה-merge/push של התוצר-הנדון. רק `{{COUNCIL_MEMBER}}`
(Council) רשאי לבטל צו-עצירה. הביטול מתועד ב-ledger.

### 5ד. ערוץ-דיווח

`oversight-report.md` עולה **ישירות ל-Council** (עוקף the orchestrator), שכן
the orchestrator מקונסולד דיווחים _וגם_ מחליט מה עולה ל-Council (לא יכול לבקר את-עצמו).
זהו ratio-העצמאות של הבקרה.

**קריטריון-מעבר:** אין צו-עצירה פעיל על התוצר. אם `{{ENABLE_OVERSIGHT}} = false` —
שער זה מדולג, ובקרת-האיכות נשענת על שערים 1-4 בלבד.

## הערות-יישום

- **{{PKG_MANAGER}}** — מנהל-החבילות של הפרויקט (למשל npm/yarn/pnpm/bun). כל פקודת-סקריפט
  רצה דרכו (`{{PKG_MANAGER}} run <script>` / `{{PKG_MANAGER}} exec <bin>`).
- **husky / hook-manager** — אופציה אחת למימוש שער 1-2. הפרויקט חופשי לבחור מנגנון אחר
  (lefthook, simple-git-hooks, או `.git/hooks` ידני). אל תתייחס לאף כלי כתלות-חובה.
- **non-fatal steps** — צעדי-עזר נלווים ב-hook (כמו רענון-נכס-נגזר) חייבים `|| true`
  כדי לא לחסום commit לגיטימי בגלל כשל-עזר.
- **{{DEFAULT_BRANCH}}** — דחיפה ישירה אליו מותרת רק תחת `{{APPROVAL_CONDITION}}`
  עם כל השערים הירוקים.
